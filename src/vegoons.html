Vegoons

Streamlined mobile web app for hungry vegans.

Live Site

Vegoons is a web app that narrows down search parameters for recipes/restaurants and returns relevant results. It started as a practice project for React js, and is based on my own process of choosing food.

Initial ux flow:

Version 1

1. Onboarding

On the first page, the app collects three pieces of data to filter results: It automatically detects location and time (to filter by breakfast/lunch/dinner), and asks the user if he's going out or staying in.

To lay the foundations of the app, I planned for three topics upfront: State management, data fetching, and a ui library. Respectively, I chose React states/props, Rest apis from Yelp and Edamam, and Fluent ui Northstar. Fluent ui allows me to easily change theme in response to user color preference.

2. Categories

Following my own preferences of choosing by cuisine, cuisine categories are the main focus of the second page, while search is presented as a secondary option.

Because the apis take different parameters, I save cuisines in a separate state from search. This lets me pass cuisines directly in the appropriate parameter for the Edamam api.

3. Results

The user-inputs up to this point – along with location, time, & an additional 'vegan' parameter – are passed to the search.

To accommodate for the Cors policy set by the Yelp api, I set up a serverless proxy api to fetch Yelp/Edamam apis, then called this proxy from the client.

The Future: Version 2

Working with imperfect data

Problem 1: In version 1, it became quickly apparent that the data points I was working with were flawed. For example, in the original version, the onboarding options of 'going out' and 'staying in' were to call different Yelp api endpoints. If the user were going out, the main endpoint would be called; if he were staying in, the endpoint returning delivery-offering-restaurants would be called. As it turned out, the delivery endpoint filtered out many restaurants whose profiles didn't explicitly list 'delivery.' However, during the pandemic, essentially every open restaurant offers delivery, so to filter some out on this basis was unfair.

To adjust for this, I programmed all Yelp searches to call the main endpoint. Now, 'going out' and 'staying in' both return the same restaurant results, making the onboarding page basically redundant.

Solution: Version 2 would remove the onboarding page.

Problem 2: The 'vegan' tag was also inadequate; for Yelp results, it can return restaurants which only have one 'vegan' item buried in the menu or reviews. Edamam accepts 'vegan' as a specific parameter to filter by; but even it returns non-vegan recipes which offer no vegan substitutes.

Considered solutions:

  1. For Yelp results, I considered creating pages for items that highlight the mention of 'vegan' in menu items and reviews. However, those aren't available as api data, and webcrawling for them would be unreliable.
  2. Another solution could have been to look for different apis, but I haven't seen any as robust as either Yelp or Edamam.
  3. If I can't add functionality to the app, the best next thing is to be honest with users about what to expect (see next point).

Making app functionality more obvious

Problem: After receiving feedback from friends, it was clear that not all of the app's filtering is obvious. For example, the app adds 'vegan' to every api call, although that is never shown to the user. It is also not obvious that recipes are sorted by course (breakfast, lunch, dinner) according to time of day.

Solutions:

  1. 'Vegan' should be shown by the search bar to make it obvious that vegan results are being searched for.
  2. Filters should be added by the search bar that say what has been applied – for example, which course applies – and allow the user to change them.

I learned a lot from this project, from making my first React web app to learning to call apis and make serverless ones. This was also my first time working with a ui library, and I was able to see how developer-friendly and efficient a css-in-js library can be.

There's still more room for improvement: For example, to make use of Fluent ui's philosophy, I would add animations that suggest functionality to the user. I could also edit the app manifest to make it a pwa. But after three weeks of hard work, I'm happy how this turned out, and am excited about the prospect of working with pwas in the future.
